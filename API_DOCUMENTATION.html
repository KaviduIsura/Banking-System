Architecture Overview

Browser / React App (http://localhost:5173) │ │ HTTPS (TLS 1.2+ with self-signed cert) ▼ FastAPI Backend (https://localhost:8443) │ ├── security/password.py ← Argon2id hashing ├── security/jwt_auth.py ← RS256 JWT tokens ├── security/mfa.py ← TOTP 2FA ├── security/field_crypto.py← AES-256-GCM encryption ├── security/tx_signing.py ← ECDSA + HMAC signing └── db.py ← MySQL connection pool │ ▼ MySQL (securebank DB)

The application implements defence in depth — multiple independent security layers so that compromising one layer does not compromise the whole system.

Endpoint Reference

1. GET /health

Purpose: Checks if the server is running.
Auth required: ❌ No
CW2 Control Point: None — utility endpoint

How it works

Returns a simple JSON object confirming the service name and status. Useful for monitoring systems and to confirm the server started correctly.

Request

bash curl -sk https://localhost:8443/health

Response ✅

json { "status": "ok", "service": "SecureBank API" }

Why it exists

Load balancers and health monitoring tools (e.g., AWS ELB, Docker healthcheck) call this endpoint to confirm the service is alive before routing traffic to it.

2. POST /register

Purpose: Creates a new user account and bank account.
Auth required: ❌ No
CW2 Control Points: G (Argon2id), I (AES-256-GCM)

How it works

  1. Checks that the email isn't already registered
  2. Hashes the password using Argon2id (OWASP recommended parameters: time_cost=2, memory_cost=19456 (~19MB), parallelism=1)
  3. Inserts the user into the users table
  4. Generates a random SB + 10-digit account number
  5. If a national_id is provided, encrypts it with AES-256-GCM before storing
  6. Creates a bank account with a £1000 welcome balance (100000 cents)
  7. Writes a REGISTER entry to the audit_log

Why Argon2id?

Argon2id is memory-hard — it requires large amounts of RAM to compute. This makes GPU/ASIC cracking attacks impractical, which is why it's preferred over bcrypt or SHA-256 for passwords.

Why AES-256-GCM for national_id?

National ID numbers are highly sensitive PII (Personally Identifiable Information). Storing them in plaintext would be a data breach risk. AES-256-GCM provides both confidentiality (nobody can read it) and integrity (tampering is detected via the GCM authentication tag).

Request

bash curl -sk -X POST https://localhost:8443/register \ -H "Content-Type: application/json" \ -d '{ "email": "alice@securebank.com", "password": "SecurePass123!", "national_id": "123456789V" }'

Response ✅ 201 Created

json { "message": "Registration successful", "account_number": "SB4771589783", "welcome_balance": "£1000.00" }

Error Responses

Code Reason
409 Conflict Email already registered
500 Database error

3. POST /login

Purpose: Step 1 of 2-factor login — verifies email and password.
Auth required: ❌ No
CW2 Control Points: A (rate limiting), G (Argon2id verify)

How it works

  1. Looks up the user by email
  2. Checks if the account is locked (too many failed attempts)
  3. Verifies the password using Argon2id constant-time comparison
  4. If wrong: increments failed_logins counter
  5. After 5 failed attempts: sets locked_until = now + 15 minutes
  6. If correct: resets failed_logins = 0, writes to audit_log, returns user_id
  7. Rate limited to 5 requests per 15 minutes per IP via SlowAPI

Why account lockout?

Without lockout, an attacker can try millions of passwords automatically (brute-force). After 5 failures the account is locked for 15 minutes — making automated attacks take years instead of minutes.

Why SlowAPI rate limiting?

IP-level rate limiting is the first line of defence (Control Point A). Even before the lockout triggers, repeated rapid requests are blocked at the network layer.

Request

bash curl -sk -X POST https://localhost:8443/login \ -H "Content-Type: application/json" \ -d '{ "email": "alice@securebank.com", "password": "SecurePass123!" }'

Response ✅ (MFA already set up)

json { "mfa_required": true, "mfa_enabled": true, "user_id": 1 }

Response ✅ (MFA not yet configured)

json { "mfa_required": true, "mfa_enabled": false, "user_id": 1 }

Error Responses

Code Reason
401 Unauthorized Wrong password
423 Locked Account locked after too many failures
429 Too Many Requests Rate limit exceeded

4. POST /mfa/setup

Purpose: Generates a TOTP secret and QR code for Google Authenticator enrolment.
Auth required: ✅ Yes (JWT Bearer token)
CW2 Control Points: G (TOTP), I (AES-256-GCM)

How it works

  1. Authenticated user calls this endpoint
  2. Generates a cryptographically random base32 TOTP secret using pyotp.random_base32()
  3. Encrypts the secret using AES-256-GCM before storing in the database
  4. Generates a otpauth:// URI (Google Authenticator format)
  5. Renders the URI as a PNG QR code
  6. Returns the QR code as a base64-encoded PNG
  7. mfa_enabled remains FALSE until confirmed (see /mfa/confirm)

Why encrypt the TOTP secret?

If the database is breached, an attacker with access to plaintext TOTP secrets could generate valid 2FA codes for every user — completely bypassing MFA. Encrypting with AES-256-GCM means the attacker also needs the encryption key (stored separately in .env/KMS).

Why base32 for the secret?

TOTP (RFC 6238) specifies base32 encoding because it only uses uppercase letters and digits — making it easier for users to type manually in the authenticator app.

Request

bash curl -sk -X POST https://localhost:8443/mfa/setup \ -H "Authorization: Bearer <your_jwt_token>"

Response ✅

json { "qr_code": "iVBORw0KGgoAAAANSUhEUgAA...", "message": "Scan this QR code with Google Authenticator, then call /mfa/confirm" }

Embed the QR in HTML: <img src="data:image/png;base64,{qr_code}">

Error Responses

Code Reason
401 Missing or invalid JWT
404 User not found

5. POST /mfa/confirm

Purpose: Confirms MFA setup by verifying the first TOTP code.
Auth required: ❌ No (uses user_id from login step 1)
CW2 Control Point: G (TOTP verification)

How it works

  1. Retrieves the encrypted TOTP secret from DB for the given user_id
  2. Decrypts it using AES-256-GCM
  3. Verifies the submitted 6-digit code using pyotp.TOTP.verify() with valid_window=1 (allows ±30 seconds clock drift)
  4. If valid: sets mfa_enabled = TRUE in the database
  5. Writes to audit_log

Why valid_window=1?

Real-world devices have clock drift. valid_window=1 accepts the current, previous, and next 30-second window — preventing false rejections while not significantly reducing security (still requires a code from the last 90 seconds max).

Request

bash curl -sk -X POST https://localhost:8443/mfa/confirm \ -H "Content-Type: application/json" \ -d '{ "user_id": 1, "code": "123456" }'

Response ✅

json { "message": "MFA enabled successfully" }

Error Responses

Code Reason
400 MFA secret not yet set up
401 Invalid TOTP code

7. POST /mfa/verify

Purpose: Step 2 of 2-factor login — verifies TOTP code and issues a JWT.
Auth required: ❌ No (uses user_id from step 1)
CW2 Control Points: E (issues JWT RS256), G (TOTP verify)

How it works

  1. Looks up user by user_id
  2. Checks mfa_enabled = TRUE
  3. Decrypts the stored TOTP secret and verifies the code
  4. If valid: issues a short-lived RS256 JWT (15-minute expiry)
  5. JWT payload: { sub: user_id, role: "customer", iat: ..., exp: ... }
  6. Signed with the RSA 2048-bit private key (keys/private.pem)
  7. Rate limited to 10 requests per minute
  8. Writes LOGIN_SUCCESS to audit_log

Why RS256 (asymmetric) instead of HS256 (symmetric)?

  • HS256 uses one shared secret — every service that verifies tokens must have it (risky)
  • RS256 uses a key pair: the private key signs, the public key verifies. Verification services only need the public key — the signing secret stays centralised. This is essential for microservice architectures.

Why 15-minute expiry?

Short-lived tokens limit the damage from a stolen token. If a JWT is intercepted, it becomes useless in at most 15 minutes.

Request

bash curl -sk -X POST https://localhost:8443/mfa/verify \ -H "Content-Type: application/json" \ -d '{ "user_id": 1, "code": "654321" }'

Response ✅

json { "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...", "token_type": "bearer" }

Use this token as: Authorization: Bearer eyJhbGci...

Error Responses

Code Reason
400 MFA not configured for this user
401 Invalid TOTP code
404 User not found

8. GET /balance

Purpose: Returns the authenticated user's account balance.
Auth required: ✅ Yes (JWT Bearer token)
CW2 Control Point: E (JWT gateway)

How it works

  1. The require_auth FastAPI dependency extracts and verifies the JWT
  2. If the token is expired, tampered, or missing → 401 Unauthorized
  3. Reads user_id from JWT sub claim
  4. Queries accounts table for that user
  5. Returns balance in cents (raw) and formatted display

Why store balance in cents (not £)?

Floating-point arithmetic is imprecise — 0.1 + 0.2 = 0.30000000000000004 in most languages. Storing as integer cents avoids rounding errors that could cause money to "disappear" from the system.

Request

bash curl -sk https://localhost:8443/balance \ -H "Authorization: Bearer <your_jwt>"

Response ✅

json { "account_number": "SB4771589783", "balance_cents": 100000, "balance_display": "£1000.00" }

Error Responses

Code Reason
401 Missing, expired, or invalid JWT
404 Account not found

9. POST /transfer

Purpose: Transfers funds between two accounts.
Auth required: ✅ Yes (JWT Bearer token)
CW2 Control Points: E (JWT), I (HMAC), J (ECDSA signing)

How it works

  1. JWT verified — sender's identity confirmed
  2. Validates amount > 0
  3. Looks up sender's account by user_id from JWT
  4. Looks up recipient's account by account_number
  5. Checks sender ≠ recipient
  6. Checks sufficient balance
  7. Signs the transaction using ECDSA P-256 private key (keys/ec_private.pem)
  8. Computes HMAC-SHA256 integrity tag using the HMAC key from .env
  9. Atomic balance update — both debit and credit in a single transaction:
  10. UPDATE accounts SET balance = balance - amount WHERE id = sender AND balance >= amount
  11. If rowcount == 0, balance changed concurrently — rejected
  12. Saves signature + HMAC tag in the transactions table
  13. Writes to audit_log
  14. Rate limited to 10 requests per minute

Why ECDSA transaction signing?

The ECDSA signature proves that a specific transaction was authorised by the backend system that holds the private key. This simulates an HSM (Hardware Security Module) sign operation — providing non-repudiation. Anyone with the public key can verify any past transaction is genuine.

Why HMAC-SHA256 as well?

ECDSA detects tampering but is slow. HMAC-SHA256 is a fast symmetric check. Together they provide: - ECDSA → non-repudiation (only the private key holder could have signed) - HMAC → fast integrity check (the record hasn't been altered in the DB)

Why sort_keys=True in JSON serialisation?

Without sorting, {"a":1,"b":2} and {"b":2,"a":1} are different byte sequences but represent the same data. Sorting guarantees the same transaction always produces the same bytes for signing/verification.

Request

bash curl -sk -X POST https://localhost:8443/transfer \ -H "Authorization: Bearer <alice_jwt>" \ -H "Content-Type: application/json" \ -d '{ "to_account_number": "SB9964885553", "amount_cents": 5000 }'

Response ✅

json { "message": "Transfer successful", "amount": "£50.00", "signature": "MEUCIF+ETyMUNke144G6hcII...==", "to_account": "SB9964885553" }

Error Responses

Code Reason
400 Insufficient funds
400 Cannot transfer to same account
400 Amount must be positive
404 Sender or recipient account not found
401 Expired or invalid JWT

10. GET /transactions

Purpose: Returns the last 20 transactions for the authenticated user.
Auth required: ✅ Yes (JWT Bearer token)
CW2 Control Point: E (JWT gateway)

How it works

  1. JWT verified — user identity confirmed
  2. Looks up user's account ID
  3. SQL JOIN query across transactions, from_account, and to_account tables
  4. Returns each transaction with: amount, date, type (credit/debit), counterparty account
  5. type is computed relative to the viewer — same transaction is "debit" for sender, "credit" for receiver

Request

bash curl -sk https://localhost:8443/transactions \ -H "Authorization: Bearer <your_jwt>"

Response ✅

json { "transactions": [ { "id": 1, "amount_cents": 5000, "amount_display": "£50.00", "type": "debit", "from_account": "SB4771589783", "to_account": "SB9964885553", "status": "completed", "created_at": "2026-07-22 01:13:48" } ] }

11. GET /admin/audit-log

Purpose: Returns the last 100 audit log entries (admin role only).
Auth required: ✅ Yes (JWT with role: "admin")
CW2 Control Point: E (role-based access control)

How it works

  1. require_admin dependency first runs require_auth (JWT check)
  2. Then checks JWT role claim == "admin"
  3. Non-admin users get 403 Forbidden
  4. Returns the last 100 rows from audit_log

What gets logged?

Event Type When
REGISTER New user registered
LOGIN_STEP1 Password verified
LOGIN_FAIL Wrong password
ACCOUNT_LOCKED Too many failed logins
LOGIN_SUCCESS Full 2FA login completed
MFA_SETUP TOTP secret generated
MFA_ENABLED MFA successfully enabled
MFA_FAIL Wrong TOTP code
TRANSFER Money sent

Why audit logging?

Audit logs are essential for: - Forensic investigation — "who did what, when, from where?" - Regulatory compliance — financial regulations (FCA, PCI-DSS) require audit trails - Intrusion detection — patterns of LOGIN_FAIL events signal brute-force attacks

Request

bash curl -sk https://localhost:8443/admin/audit-log \ -H "Authorization: Bearer <admin_jwt>"

Response ✅

json { "audit_log": [ { "id": 1, "event_type": "REGISTER", "user_id": 1, "detail": "New account created: SB4771589783", "ip_address": "127.0.0.1", "created_at": "2026-07-22 01:00:00" } ] }

Error Responses

Code Reason
401 No/invalid JWT
403 JWT valid but role is not "admin"

13. POST /dev/setup-mfa

Purpose: Generates MFA secret and QR code without needing a prior JWT — for initial enrolment.
Auth required: ❌ No (verifies password instead)
⚠️ Remove before production

How it works

Verifies email + password, generates a TOTP secret, encrypts and stores it, returns the QR code base64 AND the plaintext secret (for manual entry). After scanning, call /mfa/confirm with the user_id and first code.

Request

bash curl -sk -X POST https://localhost:8443/dev/setup-mfa \ -H "Content-Type: application/json" \ -d '{ "email": "alice@securebank.com", "password": "SecurePass123!" }'

Response ✅

json { "user_id": 1, "qr_code_base64": "iVBORw0KGgoAAAA...", "secret_plaintext": "JBSWY3DPEBLW64TMMQ======", "next_step": "Scan QR or enter secret in Google Authenticator, then POST /mfa/confirm with user_id + 6-digit code" }

Complete Login Flow

Step 1: POST /register → get account_number Step 2: POST /dev/setup-mfa → get QR code + secret Step 3: Scan QR in Google Authenticator Step 4: POST /mfa/confirm → enable MFA (with first 6-digit code) ─────────── From now on, login works as: ─────────── Step 5: POST /login → get user_id (password verified) Step 6: POST /mfa/verify → get access_token (TOTP verified) Step 7: Use "Authorization: Bearer <token>" on all protected routes

Database Schema

```sql users id, email, password_hash (Argon2id), mfa_secret_encrypted (AES-256-GCM), mfa_enabled, role, failed_logins, locked_until, created_at

accounts id, user_id, account_number (SB + 10 digits), balance_cents, national_id_encrypted (AES-256-GCM)

transactions id, from_account, to_account, amount_cents, signature (ECDSA), hmac_tag (HMAC-SHA256), status, created_at

audit_log id, event_type, user_id, detail, ip_address, created_at ```
