The application implements defence in depth — multiple independent security layers so compromising one layer does not compromise the whole system.

🔐 Security Modules

Argon2id — Control Point G

Memory-hard password hashing. Requires ~19MB RAM per hash — makes GPU/ASIC cracking impractical. OWASP recommended over bcrypt.

JWT RS256 — Control Point E

Asymmetric tokens signed with RSA 2048-bit private key, verified with public key. 15-minute expiry. API gateway for all protected routes.

TOTP MFA — Control Point G

RFC 6238 Time-based OTP. 30-second codes via Google Authenticator. TOTP secret encrypted with AES-256-GCM before storage.

AES-256-GCM — Control Points I, J

Column-level encryption for MFA secret and national ID. Provides confidentiality + integrity (GCM tag detects tampering).

ECDSA P-256 — Control Point J

Every transaction signed with EC private key. Simulates HSM signing. Non-repudiation — proves transaction was authorised by backend.

HMAC-SHA256 — Control Point I

Fast symmetric integrity tag per transaction. Combined with ECDSA: HMAC for speed, ECDSA for non-repudiation.

Rate Limiting — Control Point A

SlowAPI limits /login to 5 req/15min per IP. Account lockout after 5 failures. Combined: network + application brute-force protection.

Least-Privilege DB — Control Point H

account_service: SELECT/UPDATE on accounts only. transaction_service: SELECT/INSERT on transactions. Root not used by app code.

POST
/register
Create a new user and bank account
🔓 No Auth Control Point G — Argon2id Control Point I — AES-256-GCM

How it works

1
Check email not already in use (409 Conflict if duplicate)
2
Hash password with Argon2id (time_cost=2, memory_cost=19MB, parallelism=1)
3
Insert user into users table
4
Generate random account number SB + 10 digits
5
Encrypt national_id with AES-256-GCM if provided
6
Create account with £1000 welcome balance (100,000 cents)
7
Write REGISTER event to audit_log

Why Argon2id? It is memory-hard — each hash requires ~19MB RAM, making GPU/ASIC cracking impractical. Superior to bcrypt for modern threat models.

Why AES-256-GCM for national_id? National ID numbers are sensitive PII. GCM provides both confidentiality (encrypted) and integrity (GCM authentication tag detects any tampering).

Why store balance in cents? Floating-point arithmetic is imprecise (0.1 + 0.2 = 0.30000000000000004). Integer cents eliminate rounding errors that could cause money to disappear.

POST
/login
Step 1 of 2-factor login — verifies email and password
🔓 No Auth Control Point A — Rate Limiting Control Point G — Argon2id verify

How it works

1
Rate limited: 5 requests per 15 minutes per IP (SlowAPI)
2
Look up user by email
3
Check if locked_until > now — reject if locked
4
Verify password with Argon2id constant-time comparison
5
On failure: increment failed_logins; lock after 5 failures (15 min)
6
On success: reset counter, log LOGIN_STEP1, return user_id

Why account lockout? Prevents automated brute-force attacks. 5 failures → 15-minute lockout → reduces attack rate from billions/second to ~1 attempt per 15 minutes.

POST
/mfa/setup
Generate TOTP secret and QR code for Google Authenticator enrolment
🔒 JWT Required Control Point G — TOTP Control Point I — AES-256-GCM

How it works: Generates a cryptographically random base32 TOTP secret, encrypts it with AES-256-GCM, stores it in the database, and returns a base64-encoded QR code PNG.

Why encrypt the TOTP secret? If the database is breached, a plaintext TOTP secret would let an attacker generate valid MFA codes for every user — completely bypassing 2FA. Encryption means the attacker also needs the AES key.

Why base32? RFC 6238 (TOTP standard) specifies base32 — it uses only uppercase letters and digits, making manual entry in an authenticator app practical.

Request (header only)
curl -sk -X POST https://localhost:8443/mfa/setup \
  -H "Authorization: Bearer <jwt>"
Response ✅
{
  "qr_code": "iVBORw0KGgoAAAANSUhEUgAAAeoAAAHqAQ...",   // base64 PNG
  "message": "Scan this QR code with Google Authenticator, then call /mfa/confirm"
}
💡Embed QR in HTML: <img src="data:image/png;base64,{qr_code}">
POST
/mfa/confirm
Enable MFA after scanning the QR code — verify first TOTP code
🔓 No Auth (uses user_id) Control Point G — TOTP verify

Decrypts the stored TOTP secret and verifies the submitted 6-digit code using pyotp.TOTP.verify() with valid_window=1 (±30 seconds clock drift tolerance). On success, sets mfa_enabled = TRUE.

Why valid_window=1? Real devices have clock drift. Window of 1 accepts the current + adjacent 30-second slots — prevents false rejections without meaningfully reducing security.

Request Body
{ "user_id": 1, "code": "123456" }
Response ✅
{ "message": "MFA enabled successfully" }
POST
/mfa/confirm-auth
Enable MFA for an already-authenticated user (Dashboard flow)
🔒 JWT Required Control Point E — JWT Control Point G — TOTP

Identical to /mfa/confirm but reads user_id from the JWT sub claim instead of the request body. Used from the Dashboard when the user is already logged in.

Request Body

Response ✅

POST
/mfa/verify
Step 2 of 2-factor login — verify TOTP code and issue JWT
🔓 No Auth (uses user_id from step 1) Control Point E — issues RS256 JWT Control Point G — TOTP verify

Verifies the 6-digit TOTP code, then issues a short-lived RS256 JWT (15-minute expiry) signed with the RSA 2048-bit private key.

Why RS256 (asymmetric) instead of HS256 (symmetric)?
HS256 shares one secret key with every verification service — a breach anywhere compromises all tokens. RS256 uses a keypair: the private key signs (backend only), the public key verifies (shareable). Zero risk from distributing the public key.

Why 15-minute expiry? Limits damage from stolen tokens — a captured JWT is useless after 15 minutes.

POST
/transfer
Transfer funds between accounts — ECDSA signed + HMAC integrity
🔒 JWT Required Control Point E — JWT Control Point I — HMAC-SHA256 Control Point J — ECDSA P-256

How it works

1
JWT verified — sender identity confirmed from sub claim
2
Validate amount > 0, find sender and recipient accounts
3
Check sender ≠ recipient, check sufficient balance
4
Sign transaction data with ECDSA P-256 private key
5
Compute HMAC-SHA256 integrity tag
6
Atomic balance update: debit sender, credit recipient in one DB transaction
7
Store signature + HMAC in transactions table. Log to audit_log.

Why ECDSA? Proves the transaction was authorised by the system holding the private key (non-repudiation). Simulates an HSM signing operation — nobody can forge a signature without the EC private key.

Why also HMAC? ECDSA is powerful but slow. HMAC-SHA256 is a fast symmetric integrity check — together they provide non-repudiation (ECDSA) and fast tamper detection (HMAC).

Why sort_keys=True in JSON serialisation? Without sorting, the same transaction dict could serialise differently — breaking signature verification. Deterministic serialisation is essential for cryptographic signing.

🗄️ Database Schema

CREATE TABLE users (
  id                    INT AUTO_INCREMENT PRIMARY KEY,
  email                 VARCHAR(255) UNIQUE NOT NULL,
  password_hash         TEXT NOT NULL,                  -- Argon2id
  mfa_secret_encrypted  TEXT,                           -- AES-256-GCM encrypted TOTP secret
  mfa_enabled           BOOLEAN DEFAULT FALSE,
  role                  VARCHAR(20) DEFAULT 'customer',
  failed_logins         INT DEFAULT 0,
  locked_until          DATETIME NULL,                  -- lockout timestamp
  created_at            TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE accounts (
  id                      INT AUTO_INCREMENT PRIMARY KEY,
  user_id                 INT NOT NULL,
  account_number          VARCHAR(20) UNIQUE NOT NULL,  -- SB + 10 digits
  balance_cents           BIGINT NOT NULL DEFAULT 0,    -- stored as integer cents
  national_id_encrypted   TEXT,                         -- AES-256-GCM encrypted
  FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE transactions (
  id            INT AUTO_INCREMENT PRIMARY KEY,
  from_account  INT,
  to_account    INT,
  amount_cents  BIGINT NOT NULL,
  signature     TEXT NOT NULL,   -- ECDSA P-256 signature (base64)
  hmac_tag      TEXT NOT NULL,   -- HMAC-SHA256 integrity tag (base64)
  status        VARCHAR(20) DEFAULT 'completed',
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE audit_log (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  event_type  VARCHAR(50) NOT NULL,
  user_id     INT,
  detail      TEXT,
  ip_address  VARCHAR(45),       -- IPv4 or IPv6
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
