SecureBank - Quality Assurance Test Cases

Use this interactive document to manually verify that the SecureBank application is functioning correctly. Tick the checkbox next to a test case once it has been successfully verified.

1. Authentication & Onboarding

TC-1.01: User Registration

Verify that a new user can successfully register an account.

  1. Navigate to the Registration page.
  2. Enter valid details (Email, Passwords matching, Full Name, DOB, etc.).
  3. Submit the form.
Expected Result: User is redirected to the MFA setup screen.

TC-1.02: MFA Setup

Verify that a new user can configure Two-Factor Authentication.

  1. Scan the QR code displayed on the MFA setup screen using Authy/Google Authenticator.
  2. Enter the 6-digit code generated by the app.
  3. Submit the code.
Expected Result: MFA is verified, and the user is redirected to the Dashboard.

TC-1.03: Login & MFA Verification

Verify that an existing user must provide an MFA code to login.

  1. Logout of the current session.
  2. Login with valid email and password.
  3. Enter the correct 6-digit MFA code on the verification screen.
Expected Result: Login succeeds and the user arrives at the Dashboard.

TC-1.04: Brute-Force Lockout

Verify that the account locks after 5 failed login attempts.

  1. Attempt to login with an incorrect password 5 times in a row.
  2. Observe the UI response on the 5th failure.
Expected Result: The UI displays an "Account locked" alert with a countdown timer.

2. Dashboard & Account Management

TC-2.01: Dashboard Data Loading

Verify that the user's balance and recent transactions load correctly.

  1. Navigate to the Dashboard.
  2. Verify the Bank Card widget displays a balance and masked account number.
  3. Verify the Recent Operations list populates or shows the empty state.
Expected Result: All data matches the backend profile data.

TC-2.02: Account Freeze (Kill Switch)

Verify that a user can freeze their account.

  1. On the Dashboard, click "Freeze Account".
  2. Confirm the freeze in the modal dialog.
Expected Result: User is forcefully logged out. Attempts to log back in will fail stating the account is frozen.

3. Financial Transfers

TC-3.01: Valid Transfer Execution

Verify that a user can send money to another valid account.

  1. Navigate to the Transfer page.
  2. Enter a valid destination Account Number and Amount.
  3. Review the transfer and proceed to MFA verification.
  4. Enter a valid MFA code.
Expected Result: Transfer is marked as Successful. Balance decreases immediately.

TC-3.02: Insufficient Funds

Verify that transfers exceeding the available balance are rejected.

  1. Enter an amount greater than the current balance.
  2. Click Review Transfer.
Expected Result: UI blocks the action and shows an "Insufficient funds" error toast.

4. Transaction History

TC-4.01: History Filtering

Verify that the history filters work correctly.

  1. Navigate to the History page.
  2. Click the "Credits" and "Debits" filter pills.
Expected Result: The table updates to show only incoming (credits) or outgoing (debits) transactions.

5. Admin Operations

TC-5.01: Unfreeze User Account

Verify that an admin can unfreeze an account.

  1. Login using an Admin account.
  2. Navigate to the Audit Log / Admin panel.
  3. Find a frozen user and click "Unfreeze".
Expected Result: The user's status becomes active, and they can login again.

6. Security & Vulnerability Mitigation

TC-6.01: JWT Expiration Verification

Verify that the authentication token correctly expires.

  1. Login to generate a valid JWT token.
  2. Wait for exactly 16 minutes without performing actions (or manually manipulate the token payload expiry date).
  3. Attempt to fetch transactions or balance.
Expected Result: API rejects the request with HTTP 401 Unauthorized, and user is routed to login.

TC-6.02: Field-Level AES-256-GCM Encryption

Verify that highly sensitive fields (like National ID and MFA Secrets) are strictly encrypted at rest.

  1. Open an SQLite browser and inspect the users table directly.
  2. Locate the national_id and mfa_secret columns.
Expected Result: The data appears as a base64 encoded ciphertext string, not plain text.

TC-6.03: ECDSA Ledger Tampering Detection

Verify that the system detects manual modification of transaction records.

  1. Open an SQLite browser and locate a completed transaction in the transactions table.
  2. Manually alter the amount_cents field to a higher number.
  3. In the application, attempt to view the transaction history.
Expected Result: The ECDSA Signature / HMAC verification fails silently or flags the record as corrupted.

TC-6.04: Suspicious IP SMTP Alerts

Verify that an unrecognized IP triggers an immediate security warning.

  1. Login successfully from your standard network.
  2. Connect to a VPN or a different network (changing your public IP address).
  3. Login successfully again.
Expected Result: An SMTP email is immediately dispatched to the account owner warning of a new IP login.

TC-6.05: SQL Injection (SQLi) Defense

Verify that the login API routes are immune to SQL injection.

  1. Navigate to the Login page.
  2. Enter admin@example.com' OR '1'='1 as the email.
  3. Enter any password.
Expected Result: The login fails cleanly with an "Invalid credentials" error. The application does not crash.

TC-6.06: Cross-Site Scripting (XSS) Prevention

Verify that the transfer "Note" field escapes malicious payloads.

  1. Perform a transfer and enter <script>alert('XSS')</script> in the optional note.
  2. Navigate to the Transaction History page.
Expected Result: The script tag is displayed as raw text (sanitized/escaped) and the alert box does NOT execute.

7. Frontend & Client-Side Security Controls (Web App)

TC-7.01: Unauthenticated Route Protection

Verify that the React Router blocks unauthorized access to secure pages.

  1. Sign out of the application.
  2. Manually type /dashboard or /transfer into the URL bar and hit enter.
Expected Result: The frontend immediately redirects the user back to the /login page.

TC-7.02: Session Storage Clearing

Verify that sensitive tokens are securely wiped upon logout.

  1. Login to the application.
  2. Open browser Developer Tools (F12) -> Application -> Local Storage.
  3. Click the "Sign out" button in the app sidebar.
Expected Result: The JWT token is instantly destroyed and removed from Local Storage.

TC-7.03: Client-Side Role-Based Access Control (RBAC)

Verify that standard users cannot access Admin pages.

  1. Login as a standard Customer.
  2. Manually type /admin/audit-log into the URL bar.
Expected Result: The frontend UI prevents the render or the API rejects the fetch with a 403 Forbidden error.

TC-7.04: UI Data Obfuscation & Masking

Verify that highly sensitive data is not exposed over-the-shoulder (Shoulder Surfing defense).

  1. Navigate to the Dashboard.
  2. Observe the Credit Card widget displaying the account number.
  3. Navigate to the Login page and type a password.
Expected Result: The account number is masked (e.g., •••• •••• •••• 1234) and password fields use the dot-mask overlay.

8. Deep Dive: Account Security Flows (How to verify in the app)

TC-8.01: End-to-End Account Freeze (The Kill Switch)

Verify the manual account freeze functionality halts all outbound transfers and immediately terminates session authority.

  1. Action in App: Log in as a standard user. Scroll to the bottom of the Dashboard and click the red ⚠️ Freeze Account button.
  2. Action in App: Click Confirm Freeze in the modal that pops up. You will be instantly logged out.
  3. Verification: Attempt to log back in with your exact correct email and password.
Expected Result: The login fails immediately with an HTTP 423 error stating "Account is frozen. Please contact support."

TC-8.02: Admin Unfreeze Account Recovery

Verify that only a system administrator can restore a frozen account.

  1. Action in App: Log in using an Admin account (e.g., the primary admin generated during setup).
  2. Action in App: Click Audit Log in the left sidebar to access the Admin Panel.
  3. Action in App: Locate the frozen user in the table. Their status badge will be red and say "FROZEN".
  4. Action in App: Click the Unfreeze button next to their name.
  5. Verification: Sign out of the Admin account, and successfully log back into the previously frozen user account.
Expected Result: The account is fully restored and the user can access the Dashboard and execute transfers again.

TC-8.03: Suspicious IP Recognition (Alerting)

Verify that the backend actively tracks the last_ip and dispatches an SMTP email when the network changes.

  1. Setup: Ensure your .env file in the backend has valid SMTP credentials (e.g. Gmail App Password).
  2. Action in App: Log in to the web app normally on your Wi-Fi network. This registers your baseline IP in the database.
  3. Action in App: Disconnect your computer from Wi-Fi and connect to a Mobile Hotspot (or enable a VPN) to change your public IP address.
  4. Action in App: Refresh the app and log in again using the exact same account.
  5. Verification: Check the email inbox associated with that user account.
Expected Result: Within seconds, an email arrives titled "Security Alert: New IP Login Detected" warning the user of the new IP access.
