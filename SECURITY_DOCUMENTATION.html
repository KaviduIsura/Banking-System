SecureBank Security Documentation

This document provides a comprehensive technical overview of the SecureBank application's architecture, security mechanisms, and data structures. It is based on the active codebase implementation.

1. How the App Works (User Journey)

Registration Flow

  1. Validation & Hashing: The user submits an email, password, and optional national ID to POST /register. The system checks the users table for a duplicate email; if found, it returns a 409 Conflict.
  2. Encryption: The plaintext password is hashed using Argon2id. If a National ID is provided, it is encrypted using AES-256-GCM.
  3. Database Write: The new user is inserted into the users table (with mfa_enabled set to FALSE). A corresponding bank account is generated with a £0.00 balance and inserted into the accounts table.
  4. Audit: An audit log entry (REGISTER) is recorded. The endpoint returns the generated account number.

MFA Enrollment Flow

  1. Setup: The user calls POST /mfa/setup (or the development equivalent /dev/setup-mfa). The backend generates a cryptographically random base32 TOTP secret.
  2. Storage: The secret is encrypted with AES-256-GCM and stored in the database.
  3. Provisioning: The plaintext secret is encoded into a QR code URI and returned to the client as a base64 PNG.
  4. Confirmation: The user scans the QR code and submits the first 6-digit code to POST /mfa/confirm. The backend decrypts the stored secret, verifies the code with a 30-second sliding window, and updates mfa_enabled = TRUE.

Login Flow (Two-Step Authentication)

```mermaid sequenceDiagram participant User participant API (FastAPI) participant Database

User->>API: POST /login (email, password)
API->>Database: Fetch user & lock status
alt Account Locked
    API-->>User: 423 Locked
else Valid Credentials
    API->>Database: Reset failed logins
    API-->>User: 200 OK (mfa_required: true)
end

User->>API: POST /mfa/verify (user_id, TOTP)
API->>Database: Fetch encrypted MFA secret
API->>API: Decrypt secret (AES-256-GCM)
API->>API: Verify TOTP
alt Invalid TOTP
    API-->>User: 401 Unauthorized
else Valid TOTP
    API->>API: Generate JWT RS256
    API-->>User: 200 OK (access_token)
end

```

Balance Check

  1. Authorization: The user makes a GET /balance request containing the Authorization: Bearer <token> header.
  2. Verification: The require_auth dependency intercepts the request, decoding and verifying the JWT signature using the RSA public key.
  3. Response: If valid, the backend extracts the user_id from the token's subject (sub) claim, queries the accounts table, and returns the balance.

Transfer Flow

```mermaid sequenceDiagram participant User participant API participant DB

User->>API: POST /transfer (to_account, amount)
API->>API: JWT Auth & Rate Limit Check
API->>DB: Fetch sender & recipient accounts
alt Insufficient Funds
    API-->>User: 400 Bad Request
else Sufficient Funds
    API->>API: Create TX payload
    API->>API: ECDSA P-256 Sign Payload
    API->>API: HMAC-SHA256 Tag Payload
    API->>DB: Atomic UPDATE sender balance (-amount)
    API->>DB: Atomic UPDATE recipient balance (+amount)
    API->>DB: INSERT into transactions (signature, hmac)
    API->>DB: INSERT into audit_log
    API-->>User: 200 OK (Transfer successful)
end

```

Admin Audit Log Access

  1. Authorization Check: A user requests GET /admin/audit-log. The require_admin dependency intercepts the request, verifies the JWT, and extracts the role claim.
  2. Access Control: If the role is not "admin", it returns a 403 Forbidden.
  3. Data Retrieval: If authorised, it fetches and returns the latest 100 entries from the audit_log table.

2. Security Technology Reference Table

Technology/Algorithm Where it's used (specific file + function/endpoint) What it protects Why this choice (not a competing alternative)
Argon2id backend/security/password.py (hash_password, verify_password) User passwords at rest. Memory-hard algorithm that resists parallel GPU/ASIC cracking attacks, unlike older fast algorithms like MD5 or non-memory-hard ones like PBKDF2.
JWT (RS256) backend/security/jwt_auth.py (issue_jwt, verify_jwt) API authentication & session integrity. RS256 uses asymmetric cryptography. The private key signs tokens, while microservices only need the public key to verify them, preventing a compromised verification service from forging tokens (unlike HS256).
TOTP (HMAC-SHA1) backend/security/mfa.py (verify_totp) Authentication against credential stuffing / compromised passwords. Standardized (RFC 6238), offline generation (no SMS interception risks), and time-bound (codes expire in 30s, mitigating replay attacks).
AES-256-GCM backend/security/field_crypto.py (encrypt_field, decrypt_field) Highly sensitive database fields (MFA secrets, National IDs). Provides Authenticated Encryption with Associated Data (AEAD). Guarantees both confidentiality and integrity/tamper-detection without needing a separate MAC.
ECDSA (P-256) backend/security/tx_signing.py (sign_transaction) Transaction non-repudiation and origin authentication. Asymmetric signatures prove that the transaction was explicitly authorised by the backend holding the private key, simulating an HSM workflow.
HMAC-SHA256 backend/security/tx_signing.py (compute_hmac, verify_hmac) Transaction record integrity. A fast, symmetric integrity tag used as a secondary check to detect any modification of the transaction record in the database.
SlowAPI (Rate Limiter) backend/main.py (@limiter.limit decorators) /login (5/15m), /mfa/confirm (5/1m), /mfa/verify (10/1m), /transfer (10/1m). Mitigates brute-force credential attacks, TOTP guessing, and volumetric abuse (DDoS) on critical endpoints.
Account Lockout backend/main.py (login endpoint) User accounts (locks for 15 mins after 5 failed attempts). Prevents sustained, targeted brute-force or dictionary attacks against a specific user account.
Least-Privilege DB Users backend/schema.sql (GRANT statements at EOF) The MySQL database. Uses separate DB accounts (account_service, transaction_service) with restricted permissions (e.g., transaction_service cannot DELETE accounts or UPDATE audit logs), limiting the blast radius of a SQLi vulnerability.
IDS Monitor backend/middleware/ids_monitor.py (IDSMonitorMiddleware) Global application routing (inspects all incoming URLs). Software-level intrusion detection that uses regex signatures to detect and log SQLi, XSS, and Path Traversal attempts for security auditing.
Audit Logging backend/main.py (_audit helper function) Non-repudiation and forensic tracking. Ensures all critical actions (REGISTER, LOGIN, MFA actions, TRANSFERS) are permanently recorded with timestamps, user IDs, and IP addresses.
SCP + GPG Backups backend/scripts/backup_db.sh Database backups at rest and in transit. SCP tunnels transfer via SSH (preventing MITM attacks), and GPG AES-256 encrypts the dump before transfer so the backup server cannot read the plaintext data.

3. Organization by Security Level

Algorithm Level

  • Argon2id: Password hashing.
  • AES-256-GCM: Symmetric encryption for database fields (MFA secrets, National IDs).
  • ECDSA (P-256): Asymmetric digital signatures for transactions.
  • HMAC-SHA256: Symmetric integrity tags for transactions.
  • HMAC-SHA1: Used internally by the TOTP algorithm.

Protocol Level

  • HTTPS/TLS 1.3: The backend is configured to use self-signed TLS certificates (via Uvicorn/SSLContext) to encrypt all API traffic in transit.
  • JWT (RS256): The authentication protocol dictating how stateless authorization claims are passed and verified.
  • SCP (SSH): The protocol used for securely transferring database backups.
  • SMTPS (Port 465): Implicit TLS protocol used for sending secure transfer confirmation emails (backend/security/mail.py).

System Level

  • Firewalls / Rate-Limiting: SlowAPI applied to authentication and financial endpoints.
  • Account Lockout: Hardcoded business logic locking accounts after 5 failed logins.
  • IDS / Intrusion Detection: IDSMonitorMiddleware detecting malicious URL payloads.
  • Audit Logging: The immutable audit_log database table tracking all system events.
  • Access Control / Least-Privilege: require_admin dependency restricting endpoints, and restricted MySQL user grants in schema.sql.
  • Secure Backups: The backup_db.sh script automating encrypted off-site storage.

4. Known Limitations and Honest Scope Notes

A reviewer evaluating this prototype should note the following simplifications and gaps:

  1. Development Bypass Endpoint: The POST /dev/setup-mfa endpoint (in main.py) exists solely to allow UI integration for new accounts without an existing JWT. The docstring explicitly notes: "DELETE THIS ENDPOINT before production deployment."
  2. Secrets Management:
  3. FIELD_ENCRYPTION_KEY and HMAC_KEY are stored in plain text in a .env file. In production, these should be managed by a Key Management Service (KMS) or Hardware Security Module (HSM).
  4. backup_db.sh hardcodes the database password (DB_PASS="991371761V") instead of using a secure .my.cnf file or environment variables.
  5. IDS Implementation limitations: IDSMonitorMiddleware is a detective control, not a preventative one. It logs malicious payloads (SUSPICIOUS_REQUEST) but does not actually block or drop the request. True blocking relies entirely on the framework's routing or downstream rate-limiters. Furthermore, it only inspects the URL, not the request body.
  6. Token Revocation: The JWT implementation is stateless. There is no token blocklist implemented in the backend. When a user clicks "Sign Out" in the frontend, the token is simply dropped from client memory; a stolen token remains valid until its 15-minute expiry.
  7. Rate Limiting Scope: While /transfer is limited to 10 requests per minute, endpoints like /register and /balance lack explicit rate limit decorators, potentially leaving them open to volumetric abuse.

5. Database Schema Summary

The database (securebank) consists of four core tables:

  1. users:
  2. Stores authentication data.
  3. Encrypted: password_hash (Argon2id), mfa_secret_encrypted (AES-256-GCM).
  4. Plaintext: email, role, lock status flags, timestamp.
  5. accounts:
  6. Stores financial balances mapped 1-to-1 to users.
  7. Encrypted: national_id_encrypted (AES-256-GCM).
  8. Plaintext: account_number, balance_cents (BIGINT for precision to avoid floating-point errors).
  9. transactions:
  10. Stores the immutable ledger of money movement.
  11. Plaintext: from_account, to_account, amount_cents, status, created_at.
  12. Cryptographic proofs: signature (ECDSA base64), hmac_tag (HMAC-SHA256 base64) ensuring the record hasn't been altered.
  13. audit_log:
  14. An append-only table recording security and system events.
  15. Plaintext: event_type, user_id, detail, ip_address, created_at. No PII should be logged here in plaintext beyond what is required for forensics.
