SecureBank: System & Security Architecture

Overview: SecureBank is a production-grade digital banking and fintech simulator. It demonstrates a highly resilient, modern core banking engine equipped with stringent cryptographic controls, immutable ledger transaction signing, and an intuitive user interface.

1. System Use Cases & Target Audience

What is the use of this project?

This system serves as a secure transactional core for financial technology applications. It handles the critical path of authenticating users, managing highly sensitive PII (Personally Identifiable Information), authorising fund transfers, and maintaining a non-repudiable audit log of ledger activities.

Where can it be used?

  • Fintech Startups & Challenger Banks: As a foundational architecture for building e-wallets, peer-to-peer (P2P) payment apps, and digital-first banking services.
  • Enterprise Financial Institutions: As a reference implementation for transitioning legacy core-banking systems to modern, cryptographically sound microservices.
  • Security Audits & Education: As a comprehensive showcase of implementing OWASP (Open Worldwide Application Security Project) standards and advanced cryptography in a modern web stack.

2. System Architecture

Frontend (Client Layer)

Built with React.js and Vite. It features a modern, responsive Single Page Application (SPA) architecture utilizing React Router for navigation.

The UI relies on a robust CSS Variable design system, offering premium fintech aesthetics (glassmorphism, subtle elevations, bespoke SVG iconography) without heavy CSS frameworks.

Backend (API & Logic Layer)

Built with FastAPI (Python). It provides an asynchronous, high-performance RESTful API gateway.

The backend is entirely decoupled from the frontend, ensuring strict validation of all requests. It utilizes Uvicorn running over enforced TLS/HTTPS for secure transport.

3. Core Cryptographic Algorithms & Security Features

The system integrates security across multiple "Control Points" spanning the protocol, application, and algorithm levels.

Security Feature Algorithm / Technology Concept & Justification
Password Hashing Argon2id Argon2id is the OWASP-recommended standard. It is a memory-hard function (configured here to use 19MB of RAM per hash), making it highly resistant to GPU, ASIC, and side-channel brute-force attacks compared to older algorithms like bcrypt or PBKDF2.
Data-at-Rest Encryption AES-256-GCM Used for encrypting highly sensitive fields in the database (e.g., National ID numbers, MFA secrets). AES-256-GCM provides both Confidentiality (symmetric encryption) and Authenticated Integrity (via the GCM tag). It uses a unique 96-bit nonce per encryption to prevent ciphertext reuse.
Authentication Tokens JWT (RS256) JSON Web Tokens signed with asymmetric RSA cryptography. The backend signs tokens with a private key, and verifies them with a public key. Tokens are strictly short-lived (15 minutes) to reduce the attack window of stolen tokens.
Multi-Factor Auth (MFA) TOTP (HMAC-SHA1) Time-based One-Time Passwords adhering to RFC 6238. A 30-second rotating code is generated using the Google Authenticator/Authy standard. Prevents credential stuffing and phishing of static passwords.
Immutable Ledger Signing ECDSA (P-256) & HMAC-SHA256 To ensure financial non-repudiation, every transaction is cryptographically signed using an Elliptic Curve private key (ECDSA). A secondary HMAC-SHA256 tag is generated. If a database administrator attempts to manually alter an account balance or transaction record, the signature verification will fail, detecting the tampering instantly.

4. Advanced Threat Mitigation Strategies

Beyond standard cryptography, SecureBank utilizes application-layer controls to actively defend against ongoing attacks. These mitigations are deeply integrated into the API endpoints (main.py) and the database schema.

  • Brute-Force & Credential Stuffing Prevention (Control Point A)

    The system actively tracks failed login attempts within the users table. When an authentication attempt fails, the failed_login_attempts counter increments.

    • If the counter reaches the threshold of MAX_FAILED_LOGINS = 5, the account is immediately placed into a temporary lock state.
    • The API responds with a strict HTTP 423 Locked status code instead of a generic 401.
    • A x-locked-until HTTP header is returned, informing the frontend exactly when the lockout expires (typically 15 minutes), preventing automated bots from hammering the endpoint.
  • Suspicious Activity Alerting (Control Point B)

    During a successful login, the system compares the incoming client IP against the user's historical login IPs stored in the login_history table.

    • If a login occurs from an unrecognized or previously unseen IP address, an asynchronous background task is triggered.
    • The system integrates an SMTP service (mail.py) running over TLS port 465 to instantly dispatch a security alert email to the account owner, empowering the user to take immediate action if their credentials were stolen.
  • Account Freezing ("Kill Switch")

    In the event of a suspected compromise, users have the ability to instantly freeze their account directly from the frontend dashboard.

    • Triggering the kill switch updates the user's database status field to 'frozen'.
    • The require_auth middleware and sensitive routes (like /transfer) strictly check this status. Once frozen, the account is strictly prohibited from processing outgoing financial transfers.
    • Existing active sessions (JWTs) lose all authorization to perform mutations, stopping attackers in their tracks. Only a system administrator can unfreeze an account via the secure admin panel.
  • Deterministic JSON Serialisation

    To compute reliable ECDSA signatures for financial transactions, the data must be serialized identically every time. In Python, dictionaries are inherently unordered.

    • Before hashing and signing, the transaction payload is deterministically serialized using json.dumps(tx_data, sort_keys=True).
    • This ensures that the keys (amount, from_account, to_account, etc.) are always ordered alphabetically, preventing non-deterministic JSON ordering bugs from invalidating perfectly legitimate financial signatures during ledger verification.

5. Conclusion

SecureBank represents the gold standard for secure, resilient application development by demonstrating that security cannot be an afterthought—it must be architected into the very foundation of the system.

By enforcing Defense-in-Depth, the system mitigates singular points of failure. The combination of memory-hard password hashing (Argon2id), authenticated field-level encryption for PII (AES-256-GCM), stateless asymmetric authentication (JWT RS256), and cryptographically signed immutable ledgers (ECDSA) guarantees data confidentiality, integrity, and non-repudiation.

Ultimately, SecureBank proves that a modern fintech application can operate securely in a hostile, zero-trust environment while maintaining a seamless and premium user experience.
